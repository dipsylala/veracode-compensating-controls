# CWE-80: Improper Neutralization of Script-Related HTML Tags in a Web Page (Basic XSS)

## Overview

Occurs when user-supplied input is rendered in an HTML page without proper encoding, allowing an attacker to inject script tags or event handlers that execute in a victim's browser.

> **Note on scope:** If HTML encoding has been applied directly at the output point, the flaw should be **remediated** — a correctly encoded output would not be reported by SAST and the risk is reduced to zero. The controls below are for cases where the flaw is still reported by SAST but risk has been lowered to an acceptable level through other means — e.g. validation happens upstream in a way the scanner cannot trace, a sanitisation library handles rich HTML, the value is not user-controlled, or a template engine applies escaping invisibly to the scanner.

---

## Compensating Controls

### 1. Template Engine Auto-escaping (False Positive — Scanner Cannot See It)

The application renders output through a template engine that auto-escapes all variable interpolations by default. Veracode flags the underlying string concatenation or raw render call but cannot trace that the template layer applies encoding at runtime.

**How to confirm:** The template engine is configured with auto-escaping enabled (e.g. Razor Views, Thymeleaf `th:text`, Angular `{{ }}` binding, Jinja2 with `autoescape=True`). The flagged value reaches the page only through this auto-escaping mechanism, never via an explicit raw/unsafe output directive (e.g. `@Html.Raw()`, `| safe`, `innerHTML`).

---

### 2. Allow-list Validation Upstream

Input is validated against a strict allow-list of permitted values before it is stored or processed. By the time the value reaches the rendering layer, it can only contain characters or values that are safe to display — the scanner sees the output call but cannot trace the earlier validation gate.

**How to confirm:** Locate the validation logic that runs before the flagged value is stored or passed to the view. Confirm it uses an explicit allow-list (permitted characters, regex pattern, or enumerated set), not a block-list. Any value failing validation is rejected and never persisted or forwarded.

---

### 3. HTML Sanitisation Library (Intentional Rich Text Rendering)

The application intentionally renders user-supplied HTML (e.g. a rich text editor) and passes content through a dedicated HTML sanitisation library before output. The library strips disallowed tags and attributes while preserving safe formatting markup.

**How to confirm:** Confirm that content is processed by a sanitisation library (e.g. DOMPurify, OWASP Java HTML Sanitizer, HtmlSanitizer for .NET) with a restrictive allow-list policy before being written to the page. The library version is current and the policy does not permit `script`, `on*` attributes, or `javascript:` URIs.

---

### 4. Value Is Not User-Controlled (False Positive)

The value flagged by Veracode originates from a trusted internal source — a hard-coded constant, a system-generated value, or data from a trusted internal API — and cannot be influenced by an external attacker.

**How to confirm:** Trace the full data flow from the flagged variable back to its source. Confirm it is never derived from HTTP request parameters, headers, body, cookies, URL path segments, or any other external input. Document the source (e.g. "loaded from application config at startup", "generated by the server as a sequential integer ID").

---

### 5. Content Security Policy (Defence in Depth)

The application returns a `Content-Security-Policy` response header that prevents inline script execution. This does not prevent injection into the DOM but ensures the browser will not execute injected scripts, substantially reducing exploitability.

**How to confirm:** HTTP responses include a `Content-Security-Policy` header whose `script-src` (or `default-src`) directive does **not** include `'unsafe-inline'`. Validate using browser DevTools Network tab or `curl -I`.

---

### 6. HTTPOnly Session Cookies (Impact Reduction)

Session cookies are marked `HttpOnly`, ensuring that even if a script were injected and executed it cannot read the session token via `document.cookie`, limiting the practical impact of any exploitation.

**How to confirm:** The `Set-Cookie` header for the session cookie includes the `HttpOnly` attribute. Verify in browser DevTools > Application > Cookies.

---

## Mitigation Text Templates

These are typically 'Mitigate by Design' in Veracode.

### Template engine auto-escaping (false positive)

```text
The flagged value is rendered exclusively through [template engine, e.g. "Razor Views" / "Thymeleaf th:text bindings" / "Angular interpolation {{ }}"], which applies HTML entity encoding to all variable output by default. Veracode identifies the underlying string as a potential XSS sink but cannot statically verify the encoding applied by the template engine at runtime. The value is never passed to a raw output directive (e.g. Html.Raw, innerHTML, | safe filter). The risk of XSS exploitation through this code path is negligible.
```

### Allow-list validation upstream

```text
Input accepted at [entry point, e.g. "the POST /profile endpoint"] is validated against a strict allow-list of permitted values before storage. The validation logic [describe briefly, e.g. "restricts the field to alphanumeric characters and spaces using the regex ^[a-zA-Z0-9 ]{1,100}$"]. Any value containing characters outside this set is rejected with a 400 response and never persisted. By the time the value reaches the rendering layer, it is structurally incapable of containing script tags or event handler syntax. The residual risk of XSS exploitation through this code path is negligible.
```

### HTML sanitisation library

```text
This endpoint intentionally accepts and renders rich HTML content (e.g. from [a WYSIWYG editor]). Before output, all content is processed by [library name and version, e.g. "DOMPurify 3.1" / "OWASP Java HTML Sanitizer (policy: formatting only)"], which strips all disallowed tags — including script, iframe, and on* event handler attributes — while preserving safe formatting markup. The sanitiser is kept current and the policy does not permit script execution. The residual risk of XSS exploitation through this code path is negligible.
```

### Value from a trusted source

```text
The value identified by Veracode as a potential XSS sink originates from [describe source, e.g. "a secure controlled property file" / "a server-generated integer ID"]. This source is controlled by [describe security controls for the trusted source]. This value is not derived from any HTTP request parameter, header, body, cookie, or other external input and cannot be directly influenced by an attacker. The finding does not represent an exploitable XSS vulnerability.
```

### Allow-list + CSP + HTTPOnly (layered, strongest case)

```text
The risk from this CWE-80 finding is lowered through multiple layered controls: (1) Allow-list validation — input is restricted to [describe constraint] before storage, preventing script-bearing values from being persisted; (2) Content Security Policy — the application enforces a CSP header with no 'unsafe-inline' in script-src, so the browser will not execute inline scripts even if injection were to occur; (3) HTTPOnly session cookies — session tokens cannot be stolen via document.cookie. Together, these controls reduce the residual risk of successful XSS exploitation to an acceptable level.
```
